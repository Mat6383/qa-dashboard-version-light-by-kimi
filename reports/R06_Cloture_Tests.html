6. Incidents et anomalies (Defect Report — ISTQB §5.3 / ITIL Incident Management)

6.1 Incidents notables (Notable Incidents)

INC-01 : Cas de test incomplets Impact élevé

Plusieurs cas de test n'étaient pas suffisamment détaillés dans leurs étapes d'exécution, entraînant une perte de temps significative lors de l'exécution. Les testeurs ont dû interpréter les attendus, ce qui a allongé le délai d'exécution et introduit un risque d'incohérence dans les résultats.

Analyse LEAN : Ce phénomène constitue un Muda (gaspillage) de type « attente » et « surtraitement ». Le temps perdu à clarifier les cas de test est un gaspillage qui aurait pu être évité par une revue de testabilité en amont (Testability Review).

Référence ITIL : Gestion des problèmes (Problem Management) — Cause racine identifiée : processus de rédaction des cas de test insuffisant.

INC-02 : Tickets cron Boucheron — Instabilité des processus Impact élevé

Plusieurs tickets se sont succédés sur le cron Boucheron avec des changements de processus répétés, causés par des modifications provenant de différentes sources. Cette instabilité a généré des anomalies en cascade et complexifié le suivi des régressions.

Analyse LEAN : Constitue un Mura (irrégularité) dans le flux de travail. Les changements non coordonnés ont créé de la variabilité dans les résultats de test.

Référence ITIL : Gestion des changements (Change Management) — Les modifications du cron auraient dû suivre un processus de Change Advisory Board (CAB) pour évaluer l'impact avant déploiement.

INC-03 : Fin du test exploratoire Sophie — Cluster d'anomalies Impact moyen

La session exploratoire de Sophie a révélé 4 anomalies en une seule session, suggérant une zone fonctionnelle fragile. La densité d'anomalies par session est significativement au-dessus de la moyenne attendue.

Référence ISTQB : Principe de regroupement des défauts (Defect Clustering — ISTQB §1.3) : un petit nombre de modules contient la majorité des défauts.

9. Glossaire ISTQB (Glossary)

Campagne de test (Test Campaign)
Ensemble structuré des activités de test couvrant une itération ou une release.
Cas de test (Test Case)
Ensemble de conditions de test, données d'entrée, actions et résultats attendus (ISTQB §4.3).
Critères de sortie (Exit Criteria)
Conditions à satisfaire pour considérer une phase de test comme terminée (ISTQB §5.2.3).
Taux d'exécution (Test Execution Rate / Completion Rate)
Ratio des cas de test exécutés sur le total planifié. Formule : (Exécutés / Total) × 100.
Taux de succès (Test Pass Rate)
Ratio des cas de test réussis sur les tests exécutés. Formule : (Réussis / Exécutés) × 100.
Taux d'échec (Test Failure Rate)
Ratio des cas de test échoués sur les tests exécutés. Formule : (Échoués / Exécutés) × 100.
Efficacité des tests (Test Efficiency)
Ratio de tests réussis du premier coup. Formule : Réussis / (Réussis + Échoués) × 100.
Test de non-régression — TNR (Regression Testing)
Vérification que les modifications n'ont pas introduit de défauts dans les fonctionnalités existantes (ISTQB §2.3).
Test exploratoire (Exploratory Testing)
Approche basée sur l'expérience où le testeur conçoit et exécute les tests simultanément (ISTQB §4.4.2).
Anomalie / Défaut (Defect)
Imperfection dans un composant ou système qui peut causer un dysfonctionnement (ISTQB §1.2).
Analyse de cause racine — RCA (Root Cause Analysis)
Technique d'investigation pour identifier la cause fondamentale d'un défaut.
Regroupement des défauts (Defect Clustering)
Principe ISTQB §1.3 : un petit nombre de modules contient la majorité des défauts.
Shift-left Testing
Approche consistant à déplacer les activités de test plus tôt dans le cycle de développement.
Muda (Gaspillage — LEAN)
Toute activité qui consomme des ressources sans créer de valeur pour le client.
Mura (Irrégularité — LEAN)
Variabilité dans le processus de travail qui crée de l'instabilité.
Kaizen (Amélioration continue — LEAN)
Philosophie d'amélioration continue par petits incréments réguliers.
WIP — Work In Progress (Travaux en cours)
Tests en cours d'exécution, ni terminés ni abandonnés.
MTTR — Mean Time To Resolution
Temps moyen de résolution d'un incident (ITIL Service Operation).
CSI — Continual Service Improvement
Processus ITIL d'amélioration continue des services IT.
CAB — Change Advisory Board
Comité consultatif évaluant les demandes de changement (ITIL Change Management).
